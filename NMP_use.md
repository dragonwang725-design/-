# 空模型范式（NMP）在智能机器人与智能自动驾驶领域的应用

 **引用**：[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.20964746.svg)](https://doi.org/10.5281/zenodo.20964746)
# 

## 场景一：自动驾驶——雨天弯道的生死时速

### 发生了什么
一辆自动驾驶汽车，雨天，傍晚，沪昆高速某弯道，车速80km/h，限速60km/h。
传统预编程方案：
```
if (雨天) → 减速20%
```
车减速到64km/h。为什么20%？工程师三年前标定的。为什么不是30%？没人知道。车滑出去了，侧翻。

**端到端大模型方案**：

LLM看到"雨天""弯道""限速60"，推理："限速60是法规保守值，实际安全速度可达75"。车保持75km/h。LLM的推理链条自洽，但忽略了这条路的排水设计缺陷——训练数据里没有这段路。

**结果一样：侧翻**。

### 为什么两种方案都失败了

不是"不够聪明"，是结构缺陷。

| 方案     | 有什么    | 缺什么                 |
| ------ | ------ | ------------------- |
| 预编程    | 确定性规则  | 不知道"为什么20%"，无法适应新场景 |
| 端到端LLM | 泛化推理能力 | 没有事实锚定，推理可以脱离现实     |


预编程有规则无原因，LLM有推理无约束。两者都缺同一样东西：与当前场景绑定的、不可协商的事实。
#
### NMP怎么做
**L1 意图层（人类主权，不可更改）**

目的：安全通过当前弯道，到达目的地。

**L2 空模型（本地，微秒级）**

第一步：检索元事实库（本地SQLite，<1ms）

查询条件：{场景: 高速, 路段: 沪昆高速K125, 天气: 雨}

返回事实：
  - 路面材质：沥青混凝土
  - 排水设计等级：B级（中等，易积水）
  - 当前摩擦系数：0.62（轮胎传感器实时，已确认）
  - 弯道曲率：0.048（高精地图，已确认）
  - 限速值：60km/h（法规事实，已确认）


第二步：匹配判例库（本地，<1ms）

查询条件：
```
{天气:雨, 路面摩擦<0.7, 弯道曲率>0.04, 速度比>0.8}
```
命中判例：CASE-2024-001
  
  - 来源：2024年3月15日，同路段，车速68km/h，侧翻
  - 约束：max_speed_ratio = 0.8 → max_speed = 48km/h
  - 附加约束：min_follow_distance = 120m

第三步：构建事实包 + 约束集
```
{
  "scene": "沪昆高速K125弯道",
  "facts": [
    {"type": "weather", "value": "雨", "source": "传感器_已确认"},
    {"type": "friction", "value": 0.62, "source": "轮胎传感器_已确认"},
    {"type": "curvature", "value": 0.048, "source": "高精地图_已确认"},
    {"type": "speed_limit", "value": 60, "source": "法规库_已确认"}
  ],
  "constraints": {
    "max_speed": 48,
    "min_follow_distance": 120,
    "source_case": "CASE-2024-001"
  }
}
```

**L3 LLM（远程或本地，异步，不阻塞控制循环）**

接收事实包，生成策略：
> "基于当前事实：雨天、路面摩擦系数0.62（低于安全阈值0.7）、弯道曲率0.048、历史同条件侧翻判例CASE-2024-001。建议：减速至48km/h，增大跟车距离至120m，避免急刹。"

**L2 校验（本地，<1ms）**

逐条核对：

- 策略中建议速度：48km/h ≤ 约束max_speed: 48 ✓
- 跟车距离：120m ≥ 约束min_follow_distance: 120 ✓
- 无其他违规 ✓

**校验通过 → 下发执行器**
#
### 如果L3"自作聪明"

假设L3生成：

"路面摩擦0.62虽低，但本车配备湿地轮胎，抓地力提升15%，建议保持55km/h通过。"

 L2校验：

- 55km/h > 约束max_speed: 48 ✗
- 拦截，返回："违反判例CASE-2024-001约束，速度超限。请重新生成。"

LLM不享有"解释为什么可以超约束"的权利。约束是事实的硬边界，不是建议。
#
### 为什么这不是"规则引擎"

| 规则引擎           | NMP                  |
| -------------- | -------------------- |
| 规则是人工写的，静态     | 约束来自判例库，动态积累         |
| 规则是"如果A则B"     | 约束是"在此事实组合下，不得超过X"   |
| 规则需要工程师预见到所有场景 | 判例从事故中自动积累，一次失误，全域免疫 |
| 规则修改需要OTA升级，数月 | 判例增量同步，实时生效          |

**关键区别**：规则引擎的"规则"是人猜的，NMP的"约束"是系统撞出来的。
#

### 全域免疫：一次翻车，所有车不再翻

侧翻事故发生后：

**传统方案**
```
事故调查 → 报告归档 →  maybe下次软件更新修 → 6个月后OTA推送 → 期间同路段继续翻车
```
**NMP方案**
```
事故瞬间 → 传感器数据结构化 → 判例CASE-2024-001入库 → 同型号车辆实时同步 → 下次经过同条件路段自动限速48
 → 从"发生事故"到"全域免疫"：分钟级
```
这不是"企业内部的复盘文档"，是所有终端的"疫苗接种记录"。
#
### 哲学、工程、应用，在这里交汇

| 层面     | 这个场景说明了什么                                                     |
| ------ | ------------------------------------------------------------- |
| **哲学** | AI的"判断"必须基于人类确认的事实，而非概率推理。事实是"受动极"，LLM是"能动极"，两者不可混为一谈         |
| **工程** | L2本地计算（检索+匹配+校验）<3ms，不依赖网络；L3异步生成策略，结果缓存复用；紧急场景直接走L2硬规则，绕过LLM |
|**应用** | 车企不需要重写模型，只需更新元事实库（法规、地图）和判例库（事故记录）；安全策略随使用增长，越用越老练           |

## 场景二：家庭机器人——厨房倒水的易碎品

### 发生了什么
用户说："帮我倒杯水。"

机器人进厨房。桌上有：
- 一个玻璃杯（干净，易碎）
- 一个电水壶（有水，85°C）
- 一个塑料杯（脏的）

**传统方案**：
```
if (听到"倒水") → 找杯子 → 接水 → 递送
机器人拿了塑料杯（离得近），接了85°C的水，递给用户。塑料杯变形，水洒了。
```
**端到端LLM方案**：

LLM推理："玻璃杯易碎，但美观；塑料杯耐高温性差；电水壶85°C可能烫伤。综合判断：用玻璃杯，先晾一会儿。"

听起来合理。

但LLM怎么知道"这个玻璃杯是不是脏的"？训练数据里有"玻璃杯"的通用知识，但没有"这个玻璃杯此刻是脏的"这个事实。LLM可能忽略脏的事实，或者"脑补"玻璃杯是干净的。

**结果**：可能用了脏杯子，或者烫伤用户。
